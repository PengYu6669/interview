# 项目权衡与交接

> 这是一份面向新对话和维护者的现状说明。内容以当前代码和已通过的检查为准；“待验证”不代表已完成。

## 当前基线

- 分支：`main`
- 最近推送：`19ad3b6`，新增管理员用户指标和会话活跃度统计。
- API：FastAPI，默认本地端口 `8000`；Web：Next.js，默认端口 `3000`。
- 数据库：PostgreSQL + pgvector；Redis 负责限流、票据重放保护和协调。
- LLM：阿里云百炼 OpenAI 兼容接口，聊天模型 `qwen3.7-plus`，所有聊天请求显式 `enable_thinking=false`。
- Embedding：DashScope `text-embedding-v4`，数据库向量维度固定为 `1024`。
- OCR：百度；语音：科大讯飞。它们不是 LLM，本次迁移没有替换。
- 质量基线：后端 172 项测试通过，Ruff、mypy、Web lint/build、Compose 配置检查通过；真实 DashScope 文本和向量请求已成功。

## 关键产品边界

### 求职画像与计划

求职画像只保存用户明确确认的结构化字段；AI 只能生成草稿。周计划允许根据现实完成情况持续调整，正式写入仍由确定性应用服务完成，不由模型直接改库。

### 学习中心与题库

公共题库由管理员维护，资料导入是异步任务。模型输出必须通过 Pydantic、题量、知识点归属和原文引用校验；引用校验失败不能假装成功。向量是派生数据，切换 Embedding 后需要重建索引并重跑检索评测。

### 模拟面试与报告

面试状态、轮次、回答和报告由数据库及应用服务管理。模型只负责计划、追问、评价和报告结构化生成；报告评价保留回答证据、模型和 Prompt 版本。实时语音仍走讯飞 Provider。

## 主要 Trade-off

| 决策 | 当前选择 | 得到的收益 | 付出的代价/风险 | 重新评估触发条件 |
| --- | --- | --- | --- | --- |
| LLM 厂商 | 统一 DashScope/Qwen | 配置单一、请求协议一致、便于成本和故障观测 | 形成单厂商依赖；Qwen 与旧模型的结构化输出差异需要持续评测 | 失败率、延迟、价格或配额明显恶化 |
| 关闭思考 | 全部聊天请求 `enable_thinking=false` | 输出更快、结构化预算更稳定、避免前端显示思考片段 | 复杂推理任务可能损失质量 | 固定评测集显示质量下降且延迟预算允许恢复部分思考 |
| Embedding 维度 | 固定 1024 | 不改数据库列结构，迁移范围小 | 模型切换后旧向量不可直接比较，必须重建索引 | 检索评测长期低于门槛或模型升级 |
| 活跃用户口径 | 会话 `last_active_at`，按小时节流；Asia/Shanghai 自然日去重 | 不需要接入完整埋点，指标可解释、写入压力可控 | 只能代表登录会话活跃，不代表页面曝光或所有行为 | 需要 DAU/WAU/留存、渠道或行为漏斗时引入事件表/分析系统 |
| 管理员指标 | 总用户、今日活跃、近 7 日活跃、今日新增、管理员数 | 查询简单、隐私边界清楚、页面可快速判断运营状态 | 暂无趋势图、留存和分渠道统计 | 运营需要趋势、留存或增长归因 |
| AI 生成题库 | 后台异步生成 + 结构校验 | 用户不必等待长任务，公共内容可统一审核 | 需要任务恢复、失败重试和管理员处理异常 | 需要用户私有资料即时生成时再增加队列容量和进度体验 |
| 应用 Skill | 目录内 `skill.json` 管元数据，`SKILL.md` 管指令，rubric 独立保存 | 运行时可渐进加载，Prompt、评分和元数据职责分离 | 与 Codex Skill 规范不同，不能直接被 Codex 自动发现 | 将训练 Skill 单独发布给 Codex 或其他 Agent 时迁移到标准 Skill 包 |

## Skill 交接

项目中的 3 个 Skill 是应用运行时的训练 Skill，不是用户目录下可直接触发的 Codex Skill：

- `business-sense-coach`：业务判断、指标、估算、取舍和验证。
- `structured-expression-coach`：STAR/PREP 结构化表达、责任边界、行动证据和同题重答。
- `career-planning-coach`：根据已确认画像、题库和训练证据生成一周计划草稿。

每个目录现在包含：

- `SKILL.md`：标准 YAML 头（只有 `name`、`description`）和正文指令。
- `skill.json`：应用运行时版本、标题、训练模式和文件路径。
- `rubric.json`：结构化评分标准。

应用加载器会剥离 `SKILL.md` 的 YAML 头，只把正文传给模型。新增 Skill 时必须同时更新这三类文件，并运行 Skill Registry 测试。

## 已知限制

- 当前管理员系统日志是 Agent 工具审计面板，不是完整 APM；没有指标时序、告警、Trace 聚合和主机资源监控。
- 当前活跃统计依赖有效会话；用户打开页面但没有触发鉴权请求不会计入活跃。
- DashScope 真实请求已做最小契约验证，完整生产评测集、价格和并发配额仍需部署前确认。
- 旧 Embedding 评测结果不能直接代表迁移后的检索质量，必须完成索引重建。
- 原始简历、完整录音和完整逐字稿默认不持久化，新增留存需求需要重新评估隐私与存储边界。

## 新对话建议入口

1. 先读本文件、`docs/系统架构.md` 和 `docs/工程开发规范.md`。
2. 从 `api/src/interview_copilot/main.py`、目标路由和对应应用服务开始追调用链，不要从目录全量阅读。
3. 修改数据库字段先写 Alembic 迁移和失败路径测试。
4. 修改模型 Provider 先更新统一配置、契约测试和固定评测，再做真实最小请求。
5. 修改前端先确认 BFF 响应 Schema；不要让客户端绕过 BFF 读取服务端配置或密钥。

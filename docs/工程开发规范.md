# 工程开发规范

## 不确定事项处理顺序

1. 阅读现有代码和已安装依赖自带的文档。
2. 查询对应产品及准确版本的当前官方文档。
3. 文档仍有歧义时编写最小验证代码或契约测试。
4. 涉及产品意图、凭据、计费、隐私或不可逆选择时，询问项目负责人。

禁止用“看起来合理”的请求体填补未知信息。只有类型接口但明确未实现，也比伪造生产代码安全。

面向用户的错误、提示、状态和 AI Prompt 默认使用中文。代码标识符、OpenAPI 字段、HTTP/MIME、厂商参数和其他稳定协议不得为了中文化而改名。

重要架构决定必须在 `docs/技术决策/` 中编写 ADR，说明背景、决定、备选方案、影响和回退条件。

## 文档与目录预算

- 优先更新现有文档，禁止为一次开发过程创建总结、周报、复盘、临时方案或同义架构文档。
- 常驻文档限于入口 README、工程规则、系统架构、待确认问题，以及确有长期维护价值的 ADR、API 和运维说明。
- 新文档必须有明确的长期读者与更新触发条件；无法说明时不得创建。
- 项目掌握资料需要落盘时统一放在 `docs/项目地图.md`，默认不拆分模块卡、每日记录或假掌握清单。
- 目录只在第一个真实职责出现时创建，不使用空目录或占位文件表现“架构完整”。
- 删除失效说明或就地修正，禁止保留互相冲突的新旧版本文档。
- 中文文件名只允许出现在 `docs/` 和仓库根目录的产品文档中；代码、迁移、配置、脚本、测试、静态资源及其目录名统一使用英文 ASCII 命名。

## Python

- 支持 Python 3.12 和 3.13；生产容器固定到准确补丁版本。
- 公共函数必须提供类型标注，边界数据使用 Pydantic。
- 异步只用于网络和数据库 I/O；OCR 等 CPU 密集任务放入 Worker。
- 使用 `pyproject.toml` 管理项目，生产部署前必须锁定依赖版本。
- Ruff 负责代码规范，mypy 负责静态类型，pytest 负责测试。
- 文件上传必须限制大小、验证扩展名与真实 MIME、限制解析资源，并在生产存储前接入恶意文件检测。

## TypeScript 与 Next.js

- 保持 TypeScript 严格模式。
- 优先使用 React Server Components；只有必须使用浏览器状态或 API 时才添加 `"use client"`。
- 使用版本敏感的 Next.js API 前，按 `web/AGENTS.md` 要求阅读 `web/node_modules/next/dist/docs/`。
- 需要鉴权或数据最小化时，客户端组件不能绕过 BFF 随意直连 Python 服务。
- 服务端专用模块必须明确标记，禁止被导入客户端依赖图。
- 无障碍和响应式表现属于验收条件，不是最后补做的美化工作。

## 数据库

- 使用 UUID 主键、UTC 时间、明确外键，并根据真实查询路径创建索引。
- 使用 Alembic 保存结构变更历史；初始化 SQL 只负责本地启用数据库扩展。
- 所有查询必须参数化，禁止把用户或模型内容拼入 SQL。
- 数据访问层必须限制用户数据归属，并测试跨用户访问被拒绝。

## 第三方服务

- 每个厂商实现稳定的内部协议，并把错误转换成明确的领域错误。
- 设置连接、读取和总超时；使用有上限的退避重试、并发限制和必要的熔断。
- 只有幂等请求才能自动重试，否则必须使用幂等键。
- 凭据只来自服务端环境变量或密钥管理服务，并从日志中脱敏。
- 非显而易见的协议实现必须记录官方文档地址和核验日期。

## 完成标准

- 功能符合验收行为，不含隐藏占位符或假成功。
- 单元、集成、契约和失败路径测试按风险覆盖并通过。
- lint、类型检查、构建和 Compose 校验通过。
- 新配置写入 `.env.example`，不得包含真实值。
- 已检查安全、隐私和日志影响。
- 文档与待确认问题反映真实实现状态。

# InterviewCopilot（面壁）产品需求文档

**版本**：v3.0  
**日期**：2026-07-11  
**定位**：面向技术岗位求职者的个性化 AI 面试训练与能力提升系统  
**首要人群**：AI 应用开发方向求职者（实习、校招、社招）

---

## 一、产品定义

### 1.1 用户问题

现有工具能提供题目或对话，却很难完成一次可信、连续、可复盘的训练：

| 用户问题 | 题库 | 通用 AI 对话 | InterviewCopilot |
|---|---|---|---|
| 问题是否贴合本人经历 | 否 | 依赖用户临时描述 | 基于简历证据深挖 |
| 是否针对目标岗位 | 弱 | 不稳定 | 基于 JD 建立考察蓝图 |
| 是否连续追问 | 否 | 偶发 | 按回答漏洞和深度动态追问 |
| 评价是否有依据 | 无 | 通常只有泛化点评 | 引用原回答、评分锚点和知识证据 |
| 是否形成长期提升 | 无 | 无 | 弱项回流、复训与能力趋势 |

核心假设：

> 高质量面试训练 = 个人经历真实性 × 岗位能力匹配 × 真实面试考法 × 可验证的反馈闭环。

“三角 RAG”负责前三项的信息基础，Multi-Agent 负责考察过程中的角色制衡，Report Layer 负责把一次面试沉淀为下一次训练的输入。

### 1.2 产品原则

1. **像面试，而不是像答题**：问题之间存在上下文，面试官会听、判断、追问和切题。
2. **尖锐但不羞辱**：压力来自问题质量、时间和质疑，不来自人身评价或刻意打断。
3. **评价必须可追溯**：每个判断都应关联用户原话、评分标准及必要的知识依据。
4. **允许不知道**：诚实说明边界优于编造；系统评价应区分知识盲区、表达不清和事实错误。
5. **训练必须可迁移**：最终目标不是记住某道题，而是改善知识结构、项目叙事和临场表达。

### 1.3 核心闭环

```text
材料输入 -> 能力与岗位建模 -> 面试蓝图 -> 动态面试
   ^                                      |
   |                                      v
复训计划 <- 弱项画像 <- 证据化报告 <- 逐题评估
```

一次训练包括：

1. 上传或粘贴简历、JD，可补充目标公司、职级和面试轮次。
2. 系统展示解析结果和面试蓝图，用户可以修正错误信息。
3. 用户选择时长、模式、输入方式和重点训练范围。
4. AI 面试官按蓝图提问，并根据回答实时审计和动态追问。
5. 面试结束后生成逐题证据、能力评分、改写示范和复习路径。
6. 弱项进入能力画像，并影响下一次面试的题目和追问策略。

### 1.4 成功标准

北极星指标：**有效训练完成次数**。一次有效训练需满足约定时长或题量，并产出至少三个可证实的能力判断。

| 类别 | 指标 | 含义 |
|---|---|---|
| 使用 | 面试开始率、完成率、7/30 日复训率 | 产品是否真正进入训练习惯 |
| 训练 | 同技能复测提升、弱项收敛率 | 用户是否在变强 |
| 质量 | 人工面试官评分一致性、事实误判率 | AI 评价是否可信 |
| 体验 | 首字延迟、中断率、报告有用度 | 过程是否流畅且有价值 |

## 二、用户与场景

### 2.1 用户阶段

| 阶段 | 主要诉求 | 推荐训练 |
|---|---|---|
| 初次求职 | 不知道面试如何展开 | Relaxed 综合面 |
| 集中准备 | 补齐目标岗位知识与项目表达 | Normal 专项面/综合面 |
| 临近面试 | 还原目标公司和轮次压力 | Stress 模拟面 |
| 面后复盘 | 判断失误原因并针对性复训 | 弱项复训 |

### 2.2 面试类型

- **综合模拟**：按真实轮次覆盖自我介绍、项目、技术、系统设计、Coding 和反问。
- **专项训练**：只训练项目深挖、技术主题、系统设计、行为面或 Coding。
- **弱项复训**：根据历史报告生成新题，避免简单重复原题。
- **目标公司模拟**：在数据来源和置信度允许时匹配公司风格；禁止声称“完全还原”。

### 2.3 面试模式

| 模式 | 行为 | 反馈时机 |
|---|---|---|
| Relaxed | 允许澄清和有限提示，追问节奏较缓 | 可选题后简评 |
| Normal | 标准追问，不主动给答案 | 面试结束统一反馈 |
| Stress | 更严格的时间限制、更高质疑密度、较少提示 | 面试结束统一反馈 |

用户可随时暂停、结束或举报不合理问题。Stress 模式不得使用羞辱、嘲讽、身份攻击或无意义的频繁打断。

## 三、完整用户旅程

### 3.1 准备页

用户提供：

- 简历：PDF、DOCX、Markdown、纯文本或直接粘贴。
- JD：文本、文件或职位链接解析结果。
- 可选信息：公司、职级、面试轮次、语言、预计时长。
- 训练设置：综合/专项、Relaxed/Normal/Stress、文字/语音。

系统解析后展示可编辑结果：项目、职责、技术栈、量化指标、JD 必需项、加分项和信息缺口。任何低置信度内容都必须标记，不能静默写入面试上下文。

### 3.2 面试蓝图确认

开始前展示：

- 预计时长和阶段安排。
- 重点能力及覆盖比例。
- 简历中可能被深挖的项目。
- 不会展示具体题目和参考答案。

用户可以锁定重点、排除不相关技能，也可以直接采用系统方案。

### 3.3 面试室

界面包括面试官对话、当前阶段、剩余时间、暂停/结束、语音状态；Coding 和系统设计阶段按需展开代码板或白板。面试中不显示实时分数，避免改变用户行为。

关键状态：准备中、面试官思考、用户作答、追问、连接中断、暂停、恢复、阶段切换和提前结束。中断恢复后应明确复述当前问题，不重复计分。

### 3.4 报告页

报告按“结论 -> 证据 -> 改进 -> 复训”组织：

1. 总体表现和可信度说明。
2. 面试时间线与逐题回放。
3. 能力维度评分及原回答证据。
4. 关键错误、遗漏和表达问题。
5. 更优回答结构或示范，不伪造成唯一标准答案。
6. 分优先级的学习与复训任务。
7. 对某条评价提出异议、重新评估或排除出画像。

### 3.5 历史与能力中心

用户可以按岗位、技能和时间查看趋势；系统区分“未考察”“证据不足”和“能力较弱”，不能将未出现的能力记为零分。技能画像应支持追溯到具体场次和回答。

## 四、面试方法论

### 4.1 面试蓝图

Planner 根据简历、JD、面经、职级和时长生成结构化蓝图：

```typescript
interface InterviewBlueprint {
  targetRole: string;
  targetLevel?: string;
  durationMinutes: number;
  sections: Array<{
    phase: InterviewPhase;
    minutes: number;
    competencies: string[];
    targetDepth: 1 | 2 | 3 | 4;
    requiredEvidence: string[];
  }>;
  coverageConstraints: string[];
}
```

蓝图同时约束覆盖度和深挖程度。系统不能因为一个追问点无限延长，也不能为了走流程而忽略明显漏洞。

### 4.2 问题选择

每个候选问题计算四类信号：

- **Relevance**：与简历证据及 JD 能力的相关性。
- **Coverage**：是否补足尚未考察的关键能力。
- **Information Gain**：是否能减少对用户能力判断的不确定性。
- **Naturalness**：是否能承接上一轮回答，符合真实面试语境。

题目必须记录来源、目标能力、期望证据和难度，生成时避免把参考答案泄露在问题中。

### 4.3 追问策略

Auditor 可产生以下信号：

| 类型 | 含义 | 典型动作 |
|---|---|---|
| SHALLOW | 只有概念，没有机制或取舍 | 追问原理、边界、替代方案 |
| VAGUE | 缺少角色、动作或数据 | 要求具体例子和指标口径 |
| CONTRADICTION | 与前文或简历冲突 | 引用冲突点，请用户澄清 |
| POSSIBLE_ERROR | 疑似事实错误 | 进入事实核验后再决定是否追问 |
| EVASION | 未回答核心问题 | 重述问题，缩小范围 |
| STRONG_SIGNAL | 出现高质量证据 | 提升难度或转入新主题 |

追问采用深度预算：同一主题默认最多连续追问 3 层；遇到核心岗位能力、明显矛盾或用户主动展开时可扩展，但必须受总时长和覆盖约束。连续两次“不了解”后停止逼问，并在报告中记录知识边界。

### 4.4 阶段流转

```text
WARMUP -> PROJECT_DEEP_DIVE -> TECHNICAL -> SYSTEM_DESIGN
       -> CODING -> BEHAVIORAL -> CANDIDATE_QA -> CLOSING
```

具体场次可以跳过阶段，但顺序由蓝图显式决定。流转条件综合阶段时间预算、必考能力覆盖度、追问队列和用户状态，而不是只依赖固定题数。

### 4.5 反问环节

AI 应以目标公司的通用公开信息和虚拟面试情境回答，并明确哪些内容无法代表真实公司。报告评价反问的问题质量、岗位洞察和双向选择意识，而非评价用户是否“讨好面试官”。

## 五、三角 RAG 与知识系统

### 5.1 三类信息源

1. **Resume Layer**：用户做过什么，哪些陈述可以被追问和核验。
2. **JD Layer**：岗位要求什么，哪些能力优先级最高。
3. **Interview Corpus Layer**：真实场景通常如何发问、追问和区分水平。

三层不是简单取交集：简历决定个性化入口，JD 决定考察权重，面经决定问题形态和难度分布。系统还需保留通用基础题，以识别简历没有覆盖但岗位必需的能力。

### 5.2 数据结构

```typescript
interface EvidenceChunk {
  id: string;
  layer: "resume" | "jd" | "interview_corpus" | "knowledge";
  content: string;
  embedding: number[];
  metadata: {
    sourceId: string;
    sourceType: string;
    skills: string[];
    company?: string;
    role?: string;
    level?: string;
    difficulty?: "easy" | "medium" | "hard";
    provenance?: string;
    license?: string;
    confidence: number;
  };
}
```

面经数据必须保存来源、采集时间、授权或许可状态；用户上传内容默认仅供本人使用，不自动进入公共语料。

### 5.3 检索链路

```text
Query Construction
 -> 三层并行 Hybrid Search（Vector + FTS + metadata filter）
 -> RRF Fusion
 -> Cross-encoder/LLM Re-ranking
 -> 覆盖度与去重约束
 -> Question Candidate Set
```

检索 Query 不只来自简历和 JD，还包括当前阶段、已问问题、剩余能力缺口、历史弱项和用户上一轮回答。所有进入 Prompt 的语料都保留 citation id，用于调试和报告追溯。

### 5.4 事实核验

Auditor 不直接把模型判断写成“技术错误”。POSSIBLE_ERROR 进入 Verification 流程：

1. 提取可验证主张。
2. 检索版本匹配的权威知识库、官方文档或审核过的参考材料。
3. Verifier 输出 supported、contradicted、uncertain，并附证据和版本。
4. 只有 contradicted 且置信度达到阈值时才能计入事实错误；uncertain 只提示人工复核。

### 5.5 冷启动与质量治理

- Seed Corpus 按岗位和能力树构造，不以数量代替质量。
- 面经先去标识化、去重和人工抽检，再进入检索库。
- 对题目记录曝光、跳过、举报、区分度和评分一致性。
- 低质量、过时或来源不清的内容自动降权或下线。

## 六、Multi-Agent 架构

### 6.1 Agent 职责

| Agent | 职责 | 是否直接面对用户 |
|---|---|---|
| Planner | 生成及动态维护面试蓝图 | 否 |
| Interviewer | 提问、承接回答、控制节奏和阶段 | 是 |
| Auditor | 发现深度、矛盾、回避和强信号 | 否 |
| Verifier | 核验可验证的技术主张 | 否 |
| Evaluator | 基于完整证据生成逐题与总体评价 | 否 |
| Coach | 将弱项转为复习和复训计划 | 报告中呈现 |

Multi-Agent 的价值是职责隔离、可观测和可独立评测，不要求每轮所有 Agent 都调用。路由器根据风险和场景决定调用链，避免无意义的成本与延迟。

### 6.2 单轮协议

```typescript
interface TurnDecision {
  auditSignals: AuditSignal[];
  verifiedClaims: VerifiedClaim[];
  action: "follow_up" | "next_question" | "change_phase" | "close";
  rationale: string;
  nextQuestion?: string;
  targetCompetencies: string[];
}
```

Agent 之间只传递结构化结果。面试官不能向用户泄露内部评分、Prompt、参考答案或 Auditor 的隐式推理。

### 6.3 降级策略

- Auditor 超时：Interviewer 使用保守规则继续，不把未审计回答判错。
- Verifier 无证据：标记 uncertain，不阻塞面试主流程。
- 检索失败：使用蓝图中的通用能力题，并在链路中记录降级。
- 模型服务失败：保留 Checkpoint，允许重试或切换兼容模型。

## 七、状态机与会话

LangGraph 负责显式状态、条件路由、人工中断和恢复；PostgreSQL 保存业务事实与历史记录，Redis 可用于短期缓存和低延迟状态，不作为唯一持久化来源。

```typescript
interface InterviewState {
  sessionId: string;
  status: "preparing" | "active" | "paused" | "completed" | "aborted";
  phase: InterviewPhase;
  blueprint: InterviewBlueprint;
  currentQuestion?: Question;
  turns: InterviewTurn[];
  auditSignals: AuditSignal[];
  verifiedClaims: VerifiedClaim[];
  competencyEvidence: Record<string, CompetencyEvidence[]>;
  remainingSeconds: number;
  mode: "relaxed" | "normal" | "stress";
  version: number;
}
```

Checkpoint 在每轮回答提交、阶段切换和暂停时写入。恢复必须幂等，避免重复生成问题、重复扣时或重复计分。会话版本用于处理多标签页和网络重试造成的并发写入。

## 八、评分与报告

### 8.1 评分 Rubric

每个实际考察到的能力按 0～4 评分：

| 分数 | 锚点 |
|---|---|
| 0 | 未回答、完全偏题，或核心结论严重错误 |
| 1 | 知道名词，但无法解释机制或应用 |
| 2 | 基本正确，能描述常规做法，缺少边界与取舍 |
| 3 | 正确且有结构，能结合项目证据解释选择和结果 |
| 4 | 能处理复杂约束、反例、替代方案及量化验证 |

评价维度包括技术正确性与深度、问题分析、项目真实性证据、取舍意识、表达结构、追问适应和岗位匹配。不同面试类型采用不同维度集合与权重；未考察维度不计分。

### 8.2 证据化评价

每条评价必须包含：

```typescript
interface EvaluationItem {
  competency: string;
  score: 0 | 1 | 2 | 3 | 4;
  confidence: number;
  answerQuote: string;
  rationale: string;
  knowledgeCitations?: string[];
  improvement: string;
  recheckRecommended: boolean;
}
```

总分是实际考察能力的加权聚合，并显示覆盖度和置信区间。覆盖不足时不得输出看似精确的“综合能力结论”。等级只用于快速阅读，不能替代逐项证据。

### 8.3 报告内容

- 总结：优势、主要风险、岗位匹配结论及置信度。
- 时间线：问题、回答、追问与关键转折。
- 能力矩阵：得分、证据、覆盖度和历史变化。
- 错误分类：知识错误、知识边界、缺少证据、表达问题、策略问题。
- 改进示范：回答框架、关键遗漏和一版更优表达。
- 行动计划：P0/P1/P2 学习任务、验收方式和建议复训日期。

### 8.4 长期能力画像

能力树采用层级结构，例如：

```text
Redis
|- 数据结构与使用场景
|- 持久化
|- 内存与淘汰
|- 高可用与集群
`- 缓存一致性与故障治理
```

画像保存分数之外的证据数量、最近考察时间、难度、置信度和遗忘风险。下一次训练优先选择高岗位权重、低置信度、长期未复测或历史薄弱的节点。

## 九、语音、Coding 与系统设计

### 9.1 Voice Layer

语音模式支持流式转写、端点检测、可控打断和实时播放。报告可分析语速、停顿、口头禅、回答时长和结构，但不得从声音推断性格、情绪稳定性等缺乏可靠依据的特征。转写文本允许用户修正后再进入事实评分。

### 9.2 Coding Board

Monaco 提供题目、编辑、运行、测试用例和复杂度说明。系统记录思考过程、澄清问题、代码演进和测试行为，不只按最终是否通过评分。代码在隔离沙箱中运行，限制网络、文件、CPU、内存和执行时间。

### 9.3 System Design Board

支持组件、连线、文字和容量标注；AI 通过追问考察需求澄清、估算、数据模型、核心链路、扩展性、可靠性和取舍。画布快照和对话共同进入报告证据。

## 十、系统架构

### 10.1 建议技术栈

| 层级 | 技术 | 用途 |
|---|---|---|
| Web | Next.js、React、TypeScript、Tailwind、shadcn/ui | 用户界面与 BFF |
| Realtime | SSE/WebSocket | 流式文本、语音和会话事件 |
| Orchestration | LangGraph | 状态机、Agent 路由、Checkpoint |
| Model Gateway | Vercel AI SDK、LiteLLM 或自建 Provider Adapter | 统一不同模型厂商，OpenAI SDK 仅用于兼容端点 |
| Data | PostgreSQL、pgvector、Prisma | 业务数据、证据与向量检索 |
| Cache/Queue | Redis、BullMQ | 缓存、短期状态、异步报告任务 |
| Observability | Langfuse + 应用指标 | Prompt、模型、成本、延迟和质量追踪 |
| Sandbox | 隔离容器或受控执行服务 | Coding 运行环境 |

### 10.2 性能目标

| 指标 | 目标 |
|---|---|
| 页面交互响应 | P95 < 200ms（不含模型生成） |
| 面试官首字输出 | P95 < 2s |
| 下一问题完整生成 | P95 < 6s |
| 语音转写增量延迟 | P95 < 800ms |
| 基础报告可见 | P95 < 20s，详细报告可渐进生成 |
| Checkpoint 恢复 | P95 < 1s |

### 10.3 可观测性

每次模型调用记录 session、turn、agent、模型版本、Prompt 版本、检索 citation、延迟、token、成本和结构化输出校验结果。生产日志默认不保存原始简历和完整回答；调试采样需脱敏并获得用户授权。

## 十一、安全、隐私与内容治理

- 简历、JD、录音、转写和报告均属于用户私有数据，传输与存储加密。
- 用户可查看数据保留期限、导出记录并一键删除账号及训练数据。
- 提供“仅本次会话”“保存历史”“允许匿名改进产品”三个独立授权项，默认不用于训练公共模型。
- 自动识别手机号、邮箱、地址、证件号等敏感信息并支持脱敏。
- 面经语料需记录来源、许可和删除机制，禁止无边界抓取和复刻受保护内容。
- 防御简历、JD 和语料中的 Prompt Injection；外部内容一律视作数据而非系统指令。
- 报告明确说明 AI 评价的局限，不将结果用于真实招聘决策。

## 十二、质量评测

### 12.1 离线评测集

建立覆盖不同职级、岗位和回答质量的标准会话，由有经验的面试官标注：问题相关性、追问合理性、错误识别、评分、证据充分度和反馈可执行性。

### 12.2 核心质量门槛

| 能力 | 评测方式 |
|---|---|
| 简历/JD 解析 | 字段准确率、遗漏率、用户修正率 |
| 问题生成 | 相关性、重复率、泄露答案率、难度匹配 |
| 动态追问 | 人工合理性、无效追问率、主题失控率 |
| 事实核验 | Precision 优先，统计误判与 uncertain 比例 |
| 评分 | 与人工评分相关性、相邻档一致率、偏差分析 |
| 报告 | 证据覆盖率、用户采纳率、复训提升率 |

Prompt、模型、检索和 Rubric 的变更必须在固定评测集回归后发布。高风险的事实错误与评分漂移应支持按版本追踪。

## 十三、版本规划与验收

版本规划用于确定依赖顺序，不代表删减最终产品能力。

### Foundation：文本综合面闭环

- 简历/JD 解析与人工校正。
- 面试蓝图、三角 RAG、Interviewer、Auditor、Evaluator。
- 文本多轮面试、动态追问、暂停恢复。
- 证据化逐题报告、基础能力画像和异议入口。

验收：从材料输入到报告和复训建议完整可用；每条负面评价可定位到回答证据；会话恢复不重复计分。

### Experience：完整交互形态

- Voice Layer、Coding Board、System Design Board。
- Stress 模式和不同轮次模板。
- Verifier 权威知识核验与报告 citation。
- 历史趋势、弱项复训和间隔复习。

验收：语音、代码和画布证据均能进入统一报告；不同模式在评测集上体现稳定且可解释的行为差异。

### Intelligence：规模化质量提升

- 面经 Corpus 治理和目标公司模拟。
- 分层能力树、难度自适应和信息增益选题。
- 教练计划、复训效果评估和个性化长期路径。
- 模型路由、成本优化和系统化 A/B 测试。

验收：评分与人工面试官保持稳定一致；同技能复训可观测到显著提升；语料来源和评价链路可审计。

## 十四、风险与开放问题

| 风险 | 应对 |
|---|---|
| AI 错判技术答案 | Verifier、权威证据、置信度和用户异议 |
| 面试像机械题库 | 上下文承接、信息增益选题、自然流转评测 |
| 多 Agent 延迟和成本 | 条件路由、并行调用、缓存、降级策略 |
| 分数制造虚假精确感 | Rubric、覆盖度、置信区间和证据展示 |
| 面经版权与隐私 | 来源许可、去标识化、用户授权和删除机制 |
| Stress 模式造成伤害 | 行为红线、随时退出、禁止羞辱和敏感推断 |

仍需通过用户研究和原型测试确定：用户是否希望面试前看到蓝图、面试中允许多少提示、报告信息密度、语音打断的接受边界，以及不同职级对“高质量回答”的评分锚点。

## 十五、核心总结

InterviewCopilot 不是带简历上下文的聊天机器人，也不是把面经包装成对话。它由四个彼此约束的系统组成：

1. **三角 RAG** 连接个人经历、岗位要求和真实考法。
2. **面试策略与 Multi-Agent** 负责覆盖、深挖、核验和自然节奏。
3. **证据化报告** 让评分可解释、可质疑、可行动。
4. **长期能力画像** 让每次训练影响下一次训练，并验证用户是否真正进步。

最终目标不是让用户获得一个好看的分数，而是让用户在真实面试中更准确地理解问题、更扎实地组织证据、更清楚地表达技术判断，并在追问中守住自己的答案。

# AI 应用开发面试知识包

## 从问题到系统边界

先说明用户问题、输入、输出和质量标准，再决定是否需要 RAG、Tool Calling 或 Agent。固定步骤、没有动态分支的任务优先使用普通应用服务；只有流程存在条件分支、循环、暂停恢复或动态工具选择时，状态图或受控 Agent 才能减少复杂度。领域状态应保存在数据库，不应只存在于模型上下文或编排框架内。

## 三角 RAG 与证据隔离

候选人证据、岗位证据和公共知识证据承担不同职责。候选人语料回答“用户实际做过什么”，岗位语料回答“目标岗位需要什么”，知识语料回答“概念、实现和取舍有哪些可核验依据”。三类结果应分别召回、保留来源，再在生成阶段组合；公共知识不能补造候选人经历，简历陈述也不能自动变成技术事实。

文档进入索引前应执行字符规范化、重复页眉页脚移除、低质量 OCR 告警和结构识别。切块优先尊重标题、段落和代码块，在目标长度附近切分并保留少量重叠。切块不是越小越好：过小会丢失论证上下文，过大则降低召回精度并增加模型输入成本。

## 混合召回、阈值与重排

稠密向量适合语义近似，全文检索、关键词和 trigram 适合专有名词、代码标识符和中文短语。工程上可并行取得两路候选，再用 RRF 等与分值尺度无关的方法融合。阈值应分别应用于稠密和词法候选，不能把余弦相似度与全文排名当成同一量尺。

是否增加重排器必须由评测集决定。先记录 query、期望来源、Recall@K、MRR、引用正确率和无答案拒答率；只有基础混合召回的候选顺序确实限制效果时，才引入额外模型和延迟。多轮 query 只有在指代不清或依赖上文时才改写，明确的单轮问题不应为“显得智能”而额外调用模型。

## Tool Calling 与受控 Agent

Tool Calling 是模型返回结构化调用意图，服务端仍负责权限和执行。每个工具需要明确的输入/输出模型、用户身份注入、白名单、超时、调用预算、结果大小限制和审计。模型不能传入或覆盖 `user_id`，也不能直接获得任意 SQL、Shell、文件系统或网络能力。

Agent 的决策范围应比业务服务窄。读取证据、读取训练历史可以做成只读工具；保存回答、更新状态、扣费和删除数据应由确定性的应用服务执行。工具结果、检索文档和用户文本都是不可信数据，Prompt 中必须把指令与数据分隔，并对最终结构化输出再次校验。

## 状态、幂等与失败路径

长流程需要稳定的会话 ID、状态枚举、轮次序号和客户端幂等键。数据库是事实来源，Redis 只做缓存或协调。外部调用必须设置连接、读取和总超时；只有幂等请求才能自动重试。失败要区分用户输入错误、权限错误、厂商错误、超时和结构校验错误，不能静默降级成假成功。

## 评测与可观测性

上线前至少准备检索评测、结构化输出契约测试和端到端任务集。记录模型、Prompt、Skill、评分标准和语料版本。评价用户回答时保存逐字证据与置信度；仅凭模型判断不能把技术陈述直接定性为事实错误。生产监控应覆盖成功率、延迟分位数、Token/调用成本、工具失败率、检索无结果率和引用覆盖率。
